# Internationalization (i18n)

Pages CMS supports Sveltia/Decap-style internationalization: a repository
declares its locales and a **file structure**, collections opt in, and each
**field chooses how it localizes**. Configs without an `i18n` block are
unaffected.

## 1. Top-level configuration

```yaml
i18n:
  structure: multiple_folders   # multiple_folders | multiple_files | single_file
  locales: [en, nl, de]         # first is the default if default_locale is omitted
  default_locale: en
```

- `default_locale` must be one of `locales` (validated).
- The **default locale uses the bare path** (`hello.md`), non-default locales are
  derived from it (see structures).

## 2. File structures

For a canonical entry `content/posts/hello.md` with `locales: [en, nl]`,
`default_locale: en`:

| structure | en (default) | nl |
| --- | --- | --- |
| `multiple_folders` | `content/posts/hello.md` | `content/posts/nl/hello.md` |
| `multiple_files` | `content/posts/hello.md` | `content/posts/hello.nl.md` |
| `single_file` | `content/posts/hello.md` (locale-keyed object, both locales) | — |

`single_file` example on disk:

```yaml
---
slug: hello          # duplicate/none fields at the root (shared)
en:
  title: Hello
  body: ...
nl:
  title: Hallo
  body: ...
---
```

## 3. Per-collection opt-in

```yaml
content:
  - name: posts
    type: collection
    path: content/posts
    i18n: true        # inherit repo i18n; false to opt a single entry out
    fields: [...]
```

`i18n: true|false` on the entry wins; otherwise it inherits "is a top-level
`i18n` block present".

## 4. Per-field localization

Add an optional `i18n` key to any field:

```yaml
fields:
  - { name: title,   type: string,    i18n: translate }  # per-locale (alias: true)
  - { name: body,    type: rich-text, i18n: translate }
  - { name: date,    type: date,      i18n: duplicate }  # one value, mirrored to all locales
  - { name: slug,    type: string,    i18n: duplicate }
  - { name: id,      type: uuid,      i18n: none }        # default-locale file only
  - { name: summary, type: text }                         # omitted ⇒ none
```

| `i18n` | Editable | Stored |
| --- | --- | --- |
| `translate` / `true` | every locale pane | each locale |
| `duplicate` | default pane | every locale, identical |
| `none` / omitted / `false` | default pane | default locale only |

The markdown `body` is always treated as translatable. Fields not declared in
the schema are kept per-locale so no data is dropped.

## 5. Editor behaviour

- The locale switcher (top-right) chooses the active locale.
- In a **non-default** locale pane, only `translate` fields are shown;
  `duplicate`/`none` fields are edited in the **default** pane.
- Fields show a badge: **Per-locale** (`translate`) or **Shared** (`duplicate`).
- Saving writes **every locale file in one commit**. Untranslated non-default
  locales don't block the save (only the default locale fully validates).
- Renaming (via the default-locale filename) moves **all** locale files in one
  commit; deleting removes **all** locale files in one commit.

## 6. Collection list

Per-locale sibling files are collapsed into their canonical entry, which is
badged with the locales that exist (translated) vs. missing (dimmed). For
`multiple_folders`/`multiple_files` this is derived from the file listing.

## 7. The inline `localized-string` field

The field type `localized-string` (legacy alias `i18n`) stores
`{ en: "…", nl: "…" }` inside a single field's value. It's an escape hatch for
localizing one field in an otherwise non-i18n entry. It cannot be combined with
field-level `i18n: translate` on the same field (validated). For most cases the
structure + field `i18n` model above is preferred.

## 8. Known limitations (v1)

- **Side-by-side multi-column editing** is scaffolded (`viewMode` in the locale
  context) but not yet surfaced; editing is one locale at a time via the switcher.
- **`single_file` collection-list badges** can't tell which locale blocks are
  filled from the path alone (the body holds the truth); only presence of the
  file is reflected.
- **`single_file` load** parses the raw body client-side, so field-level *read*
  transforms (e.g. image path rewrites) are not applied for that structure.
- **Nested field recursion**: localization modes apply at the top level; a
  `duplicate` object copies its whole subtree, a `translate` object keeps its
  children per-locale.
- **Renaming from the collection list** renames only the single file; rename i18n
  entries from the editor (default locale) to move every locale.
- `enforce_all_locales`, `initial_locales`, and `canonical_slug` from the spec
  are not yet implemented.
